# Repository Guidelines

## Project Overview

OpenSEO is a TypeScript SEO application built with TanStack Start, React 19, Vite, Cloudflare Workers, Cloudflare D1, KV, R2, and Workflows. The root app is the product experience. The `web/` directory is a separate marketing/docs site with its own package, lockfile, and build.

Use `pnpm` for all Node tasks. The root package currently targets `pnpm@10.30.1`.

## Repository Layout

- `src/routes/`: TanStack Router file routes for the product app.
- `src/client/`: client UI, shared components, hooks, feature screens, layout, styles, and client-only helpers.
- `src/serverFunctions/`: TanStack Start server functions. Keep these thin; they should validate/marshal requests and delegate domain work.
- `src/server/features/`: server-side domain services and repositories grouped by product area.
- `src/server/lib/`: integrations and infrastructure helpers, including DataForSEO, R2, PostHog, and audit utilities.
- `src/server/mcp/`: OpenSEO MCP server, OAuth handling, tools, schemas, and transport tests.
- `src/server/workflows/`: Cloudflare Workflow entrypoints and workflow-specific helpers.
- `src/db/`: Drizzle schema and generated Better Auth schema.
- `drizzle/`: D1 migration files and Drizzle metadata snapshots.
- `docs/`: public and maintainer documentation that is too detailed for the README.
- `adr/`: architecture decision records.
- `release-notes/`: finalized release notes, one Markdown file per tag.
- `scripts/`: project maintenance and profiling scripts.
- `web/`: standalone marketing/docs site.

See `docs/PROJECT_STRUCTURE.md` for a more detailed map.

## Development Commands

Run these from the repository root unless noted otherwise.

```sh
pnpm install
pnpm run db:migrate:local
pnpm run dev
```

Agent-friendly dev server:

```sh
pnpm run dev:agents
```

This writes logs to `.logs/dev-server.log` and uses portless. By default the app is available at `http://open-seo.localhost:1355`.

Important checks:

```sh
pnpm run format:check
pnpm run knip
pnpm run types:check
pnpm run lint
pnpm run test:ci
pnpm run ci:check
```

Website checks:

```sh
pnpm --dir web install
pnpm --dir web run types:check
pnpm --dir web run build
```

## Coding Rules

- Prefer existing feature boundaries over new top-level folders.
- Keep server functions small and push business logic into `src/server/features/*`.
- Keep reusable UI primitives in `src/client/components`; keep feature-specific UI under `src/client/features/<feature>`.
- Keep shared validation schemas in `src/types/schemas` when both client and server use them.
- Use the `@/*` alias for imports from `src`.
- Do not edit `src/routeTree.gen.ts` or `web/src/routeTree.gen.ts` by hand; they are generated by TanStack Router.
- Do not hand-edit `worker-configuration.d.ts` or `web/worker-configuration.d.ts`; regenerate Cloudflare types with `pnpm run cf-typegen` when root bindings change, or with the web package's typegen command if one is added later.
- Do not hand-edit `src/db/better-auth-schema.ts`; regenerate it with `pnpm run auth:generate` when Better Auth schema changes.
- Add or update focused tests for behavior changes, especially in shared utilities, server services, billing, auth, MCP, and DataForSEO parsing/cost logic.

## Database And Cloudflare

- Schema changes start in `src/db/app.schema.ts` or the generated Better Auth schema.
- Generate migrations with `pnpm run db:generate`.
- Apply local D1 migrations with `pnpm run db:migrate:local`.
- Apply production migrations with `pnpm run db:migrate:prod`.
- Cloudflare bindings live in `wrangler.jsonc`; keep `Env` declarations in sync when bindings change.
- Docker self-hosting uses `AUTH_MODE=local_noauth`; do not assume it is safe on the public internet without an auth-protected proxy or private network.

## Environment

Copy `.env.example` to `.env.local` for local app development or `.env` for Docker self-hosting.

Required for SEO data:

- `DATAFORSEO_API_KEY`: base64-encoded `login:password`

Auth modes:

- `cloudflare_access`: default for deployed self-hosting behind Cloudflare Access.
- `local_noauth`: trusted local mode used by local dev and Docker self-hosting.
- `hosted`: Better Auth email/password mode for the managed app.

## Pull Request Checklist

Before handing off a code change, run the narrowest useful checks plus broader checks when shared code changed:

```sh
pnpm run format:check
pnpm run types:check
pnpm run lint
pnpm run test:ci
pnpm run verify:generated
pnpm run verify:bindings
pnpm run release:verify
pnpm run smoke:all
pnpm run billing:verify
```

Use `pnpm run ci:check` for the same static checks CI runs on the root app. If the change touches `web/`, also run `pnpm --dir web run types:check`, `pnpm --dir web run build`, and `pnpm --dir web run cf-typegen` when web Worker bindings change.

For a full pre-release local gate, run `pnpm run verify:all`.

## Release Notes

The current version is stored in `package.json`. Generate release note drafts with:

```sh
pnpm run release:notes
```

Final notes belong in `release-notes/v<version>.md`. Verify release readiness with `pnpm run release:verify`. Maintainer details are in `docs/MAINTAINERS.md`.
